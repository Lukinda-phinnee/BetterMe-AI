## Plan: AI auto-rename at 3rd message + manual rename in goals page

### Decisions (confirmed with user)
- **Trigger:** generate the title once, at the **3rd user message**, using the first 3 messages as context. Single shot.
- **No overwrite:** once a title is set (auto *or* manual), the AI never changes it again.
- **Tracking:** add a DB column to distinguish locked titles.

---

### 1. Schema — add `title_locked` column

**`packages/db/migrations/004_add_title_locked_to_conversations.sql`** (new)
```sql
ALTER TABLE chat_conversations
  ADD COLUMN IF NOT EXISTS title_locked BOOLEAN NOT NULL DEFAULT FALSE;
```

**`apps/backend/supabase-schema.sql`** — add `title_locked BOOLEAN NOT NULL DEFAULT FALSE` to the `chat_conversations` table definition (line ~192), so fresh installs match.

A boolean flag is the minimal, self-documenting way to enforce "never overwrite": auto-title only runs when `title_locked = false`; both auto-title and manual rename set it `true`.

---

### 2. Backend — `apps/backend/src/routes/ai.ts`

**a) Extract a reusable helper** `generateConversationTitle(conversationId, userId): Promise<string | null)` containing the existing Groq title logic (system prompt, `__SKIP__` handling, `max_tokens: 20`, truncate to 80 chars). Returns `null` when the model says `__SKIP__` or the call fails — callers treat null as "leave the title alone".

**b) Refactor the existing `POST /conversations/:id/generate-title`** endpoint (lines 553–645) to call the helper. Preserves its current response shape (`{ title, skipped?, conversation? }`) and ownership check — just dedups the inline Groq call.

**c) Wire auto-title into `POST /coaching`** (the endpoint the goals page uses). After the assistant reply is persisted (~line 313):
  - Count `chat_messages` where `role = 'user'` for this conversation.
  - If `count === 3` **and** the conversation's `title_locked === false`: call the helper; if it returns a title, `update({ title, title_locked: true })`.
  - Gate is `=== 3` (single shot) per the user's choice; if the 3rd message is too vague the title stays the default.

**d) Lock on WOOP-wish titles** — the existing branch that sets the title to `parsed.wish` (lines 293–298) also sets `title_locked: true`, so a WOOP-derived title isn't later clobbered at message 3.

No change to the response shape — the goals page already calls `fetchConversations()` after every exchange, so the new title appears in the sidebar via that existing refresh.

---

### 3. Frontend — manual rename in `apps/frontend/app/dashboard/goals/page.tsx`

Port the pattern already proven in `chat-history-sidebar.tsx`, adapted to the goals page's class-based styling (not inline styles):

- **State:** add `editingId` (`string | null`) and `editTitle` (`string`).
- **`handleRename(conv)`:** `PUT /api/ai/conversations/:id` with `{ title: editTitle }` using `authToken` from `useDashboard()` (not localStorage); on success update local `conversations` state and clear `editingId`. Backend's `title_locked` flag gets set there too (see below).
- **Rewrite the conversation list item JSX** (lines 358–385): when `editingId === conv.id`, render an auto-focus text input + confirm check button (Enter saves, Escape cancels); otherwise render title/time + a hover-revealed **rename** pencil button alongside the existing **delete** button.

**Backend `PUT /conversations/:id`** (lines 479–499): add `title_locked: true` to the update so a manual rename prevents any future auto-title (the user's "never overwrite" rule, applied to manual edits).
