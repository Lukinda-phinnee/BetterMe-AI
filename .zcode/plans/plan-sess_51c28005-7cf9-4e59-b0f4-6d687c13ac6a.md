## Two changes: numbered guided suggestions + rich WOOP tasks

### 1. Guided suggestions → numbered list with descriptions

**Backend prompt (`woopCoachPrompt`):**
- Change the `<GUIDED_QUESTION>` `suggestions` schema from an array of plain strings to an array of `{ "title", "description" }` objects. The prompt instructs the AI to give each option a short title and a 1-sentence explanation.

**Backend parser (`/coaching` handler):**
- Update the `<GUIDED_QUESTION>` extraction to validate the new shape. Accept both old-format (string) and new-format (object) for forward compat, but normalize everything to `{ title, description }`.

**Frontend panel JSX:**
- Replace the `.goal-chat-guided-chips` row of pills with a numbered vertical list. Each item shows a number badge, the title (clickable, fills draft), and a muted description below. Clicking any item fills the guided draft with the title text.

**Frontend SCSS:**
- Add `.goal-chat-guided-list`, `.goal-chat-guided-list-item`, `.goal-chat-guided-list-number`, `.goal-chat-guided-list-title`, `.goal-chat-guided-list-desc` — styled to match the panel's dark-slate / white theme. Number badges are indigo pills. Selected item gets a highlight border.

### 2. WOOP tasks → full add-task-form fields

**Backend prompt (`woopCoachPrompt`):**
- Extend the `<WOOP_DATA>` task schema to include: `due_date` (ISO 8601 date string), `labels` (category name), `checklist` (array of subtask strings), `color` (hex color matching column convention). 
- Add a rule: before emitting `<WOOP_DATA>`, the AI must deeply analyze the wish + outcome + obstacle + plan and create tasks that are **specific to this user's exact situation** — not generic advice. Tasks must have realistic due dates (spread over days/weeks based on the goal timeframe), appropriate categories, and actionable subtasks.

**Backend parser (`/coaching` handler):**
- Extend the `cardsToInsert` mapping to pass through `due_date`, `labels` (wrapped as `[{ name, type: "category" }]` matching the existing POST /api/cards pattern), `checklist` (wrapped as `[{ id, text, completed: false }]`), and `color`. Parse labels and checklist from string format if needed (same logic as the existing cards POST handler).

**No frontend changes for this part** — the goal page already fetches and renders cards from the DB with all their fields. The richer data just flows through.
